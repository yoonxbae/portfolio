AI 챗봇이 도입되어 있음에도 챗봇이 어떤 유형의 문의에 제대로 답하지 못하는지 데이터로 확인되지 않았기 때문에 품질 개선 역시 체계적으로 이루어지지 않았습니다.

문제 유형 파악: 쌓인 데이터를 분석하면서 챗봇이 응답하지 못하거나 잘못 응답하는 문의 유형을 분류하고, 반복적으로 나타나는 패턴을 식별해 개선 우선순위를 정했습니다.

개선 실행: AI 상담 담당 업체와 소통하며 챗봇 응답 품질을 개선하고, 채널톡 워크플로우를 직접 수정해 자주 묻는 유형에 대한 자동 응답 흐름을 최적화했습니다.

Q3. 프로젝트를 진행하며 무엇을 배우셨나요?

데이터가 쌓이는 구조를 먼저 만들면, 그 이후의 의사결정이 훨씬 빨라진다는 것을 직접 체감했습니다. 데이터가 없을 때는 "느낌"으로 판단했지만, 구조가 생기고 나서는 "이 유형이 이번 달 전체 문의의 30%"라는 명확한 근거로 우선순위를 정할 수 있었습니다.

외부 업체와의 협업에서는, 요청의 구체성이 결과 품질을 결정한다는 것을 배웠습니다. 데이터로 근거를 제시하고 구체적인 사례를 들어 요청할수록 개선 속도와 품질이 달라졌습니다.

Q4. 가장 아쉬웠던 판단은 무엇이고, 다시 시도한다면 어떻게 진행할까요?

다시 한다면 수집 → 분석 → 의사결정으로 이어지는 전체 흐름을 먼저 설계하고, 역방향으로 수집 스키마를 정의했을 것입니다.

특히 미반납 알림톡 발송 차수 산정이 수작업에 의존하고 있어 누락 리스크가 크고 반복 공수가 상당했습니다. "지금 몇 대 남아있어요?"라는 단순한 질문 하나에도 즉각 답하기 어려운 상태였습니다.

Q2. 해결 과정에서 어떤 고민이나 시도를 하셨나요?

5탭 대시보드 설계: MySQL 기반으로 재고 현황 / 출고 현황 / 입고 현황 / 미반납 현황 / 종합 현황 5개 탭을 구성했습니다. stock, stock_history, rent_device, delivery 등 테이블을 연결하는 쿼리를 직접 설계했고, 매일 00시 자동 갱신 및 매주 월·수·금 Slack 자동 리포트를 연동했습니다.

입고 로직 정의: 단순히 상태가 바뀐 것을 입고로 보는 게 아니라, "출고 → 입고대기 → 정상", "출고 → 정상" 등 의미 있는 경로만을 입고로 판단하는 로직을 7개 케이스(A~G)로 명확히 정의하고, 중복/노이즈 케이스(H~J)를 필터링했습니다.

미반납 팔로업 자동화: 학생은 마지막 수업일 기준, 선생님은 마지막 활동일 기준으로 5일 이상 경과 시 자동으로 대상자를 추출하고, 알림톡 발송 차수(1~3차 → 결제필요/내용증명)를 자동 산정하도록 구조를 설계했습니다. Zapier로 반납확인서 제출 / 회수요청 팔로업 시트도 자동화했습니다.

운영 자동화는 기술적 구현보다 '운영하는 사람이 어떻게 쓸 것인가'를 먼저 설계하는 것이 중요하다는 것을 배웠습니다. 시스템이 아무리 정교해도 입력이 불편하면 결국 수기로 돌아가게 됩니다.

또한 엣지 케이스를 얼마나 촘촘하게 정의하느냐가 데이터 신뢰도를 결정한다는 것을 깨달았습니다. 입고 로직 케이스 정의 과정에서, 경계 조건을 명확히 하지 않으면 집계 결과가 기대와 달라지는 문제가 반복됐습니다.

Q4. 가장 아쉬웠던 판단은 무엇이고, 다시 시도한다면 어떻게 진행할까요?

배포 전 실제 운영자와 함께하는 테스트 세션을 충분히 진행하지 못했습니다. 배포 이후 "이 케이스는 어떻게 처리하나요?"라는 질문이 반복됐고, 이는 설계 단계에서 운영자의 실제 워크플로우를 더 깊이 이해하지 못한 결과였습니다.

다시 한다면 파일럿 운영 기간을 먼저 설정하고, 실제 담당자의 피드백 루프를 설계 단계에 통합했을 것입니다.

Q2. 해결 과정에서 어떤 고민이나 시도를 하셨나요?

대규모 품질 진단: 약 3만 개의 질의어를 트래픽 규모(Top/Middle/Bottom), CTR/CVR 성과, 카테고리별로 다각도 분석해 우선순위를 정하고 단계적으로 품질을 개선했습니다.

품질 평가 툴: 검색 결과를 5점/3점/1점으로 평가하는 기준을 정의하고, 파트타이머가 일관된 기준으로 평가할 수 있도록 평가 툴과 운영 가이드를 설계했습니다. 수집된 평가지표는 엔지니어링 팀의 랭킹 개선에 기초 자료로 활용되었습니다.

Q3. 프로젝트를 진행하며 무엇을 배우셨나요?

운영 체계는 '만드는 것'보다 '지속하는 것'이 더 어렵다는 것을 배웠습니다. 모니터링 기준을 세우고 대시보드를 만드는 것은 시작이고, 팀 전체가 그것을 정기적으로 활용할 수 있도록 운영 루틴과 문서를 함께 설계해야 비로소 체계로 정착됩니다.

또한 운영 데이터를 단순 기록이 아닌 엔지니어링 팀의 개선 작업에 연결하는 과정에서, 운영과 개발의 협업 구조를 설계하는 역할의 중요성을 실감했습니다.

Q4. 가장 아쉬웠던 판단은 무엇이고, 다시 시도한다면 어떻게 진행할까요?

품질 평가 툴 운영 과정에서, 평가자 간 일관성을 확보하는 데 초기에 충분한 시간을 쏟지 못했습니다. 기준을 문서화했음에도 평가자마다 해석이 달라 일부 데이터를 재평가해야 했습니다.

다시 한다면 운영 시작 전 calibration session을 먼저 진행하고, 평가자 간 일치도(inter-rater reliability)를 측정하는 기준도 함께 정의했을 것입니다.
